# cr P0 notes

This file records the current product judgement for the lightweight review workflow.

## Current P0

1. Changed-file hierarchy
   - Status: implemented.
   - Why: before reading a file, the reviewer needs to know where it sits in the repository.
   - Surface: `cr diff` and `cr review` both start with a changed-file tree.
2. Per-file change hunks with context
   - Status: implemented.
   - Why: line counts and symbols are not enough to answer "what actually changed in this file?"
   - Surface: `cr review` prints compact `changes:` hunks for each changed file.
3. Compact file-purpose hint
   - Status: implemented.
   - Why: human reviewers need a quick answer to "what is this file roughly for?" before reading details.
   - Surface: `cr outline` and code-file sections in `cr review` print `purpose:`.
4. Review filters
   - Status: implemented.
   - Why: noisy working trees make terminal output hard to scan; reviewers often narrow to code or one directory first.
   - Surface: `cr diff` and `cr review` support `--code` and optional path filters.
5. Deleted/renamed file display polish
   - Status: implemented.
   - Why: file lifecycle changes are easy to misread if they look like ordinary modified files.
   - Surface: deleted files are marked in `diff`/`review`; renamed summaries preserve the old path when Git reports a rename.
6. Staged changes support
   - Status: implemented.
   - Why: reviewers often stage a subset of AI changes before review; `cr` should not miss index changes.
   - Surface: `cr diff --staged` and `cr review --staged` inspect staged/index changes and compose with `--code` and path filters.
7. Large-change summary ergonomics
   - Status: implemented.
   - Why: humans scan before reading; large reviews need totals and one-line file focus before full detail output.
   - Surface: `cr review` starts with a compact `Summary:` table.
8. Output depth controls
   - Status: implemented.
   - Why: large reviews often need a quick scan before reading full hunks.
   - Surface: `cr review --summary` shows summary/tree only; `cr review --no-hunks` keeps purpose/modified/outline without diff hunks.
9. Machine-readable review output
   - Status: implemented.
   - Why: review facts should be reusable by scripts, AI prompts, and other terminal tools without scraping human text.
   - Surface: `cr review --json` emits structured summary and file data, and composes with `--summary`, `--no-hunks`, `--code`, and `--staged`.
10. Mixed staged/unstaged awareness
   - Status: implemented.
   - Why: reviewers can easily miss index changes when reading unstaged changes, or vice versa.
   - Surface: `cr diff` and `cr review` print a note when the other side has matching changes; JSON includes `other_changes`.
11. Configurable hunk context
   - Status: implemented.
   - Why: compact hunks are good defaults, but some changes need either zero context for AI prompt budget or wider context for human reasoning.
   - Surface: `cr review --context N` controls per-file hunk context for both terminal and JSON output, defaulting to 2 lines.
12. Combined local-change view
   - Status: implemented.
   - Why: staged and unstaged changes often belong to the same review session; forcing separate reads slows down daily use.
   - Surface: `cr diff --all` and `cr review --all` use `git diff HEAD` to show staged and unstaged tracked changes together, composing with `--code`, path filters, `--json`, and hunk controls.
13. First changed-line anchors
   - Status: implemented.
   - Why: after scanning a terminal review, humans often want to jump directly to the changed place without searching inside the file.
   - Surface: `cr review` summary rows include `path:line` anchors, file-tree leaves include `line N`, detail headers include `@ path:line`, and JSON includes `anchor` plus `first_changed_line`.
14. Untracked file awareness
   - Status: implemented.
   - Why: AI coding often creates new files before they are staged; hiding them makes the review incomplete.
   - Surface: default `cr diff`/`cr review` and `--all` include untracked files from `git ls-files --others --exclude-standard`; text files are shown as full-file additions and JSON reports `status: "untracked"`.
15. Binary and large-file guardrails
   - Status: implemented.
   - Why: generated assets, archives, and oversized text files can flood terminal review output and make real code changes harder to see.
   - Surface: binary, non-UTF-8, and over-200 KB untracked files stay visible in summaries but inline hunk content is replaced with a short omitted-content note.
16. Path-based risk hints
   - Status: implemented.
   - Why: lockfiles, config files, and generated outputs are easy to skim past but often need deliberate human confirmation.
   - Surface: `cr review` summary/tree/details and JSON expose conservative `lockfile`, `config`, and `generated` hints based on changed paths.
17. Line-numbered hunk rendering
   - Status: implemented.
   - Why: reviewers often need to quote or discuss a specific changed line after reading terminal output.
   - Surface: `cr review` hunk rows include old/new line-number columns while preserving the original `+`, `-`, and context text.
18. Installable command entry point
   - Status: implemented.
   - Why: a daily CLI tool should be runnable as `cr`, not only as `PYTHONPATH=src python3 -m cr`.
   - Surface: `setup.py` exposes the `cr` console script at `cr.cli:main`, avoids isolated build dependency downloads for older pip editable installs, and README install instructions verify `cr --help`.
19. Baseline comparison mode
   - Status: implemented.
   - Why: reviewers often need to inspect an already-committed branch against `main` or another known ref, not only unstaged local edits.
   - Surface: `cr diff --base REF` and `cr review --base REF` reuse the same summary, anchors, hunk rendering, code filters, and JSON output against a named branch or commit.
20. Explicit two-ref comparison mode
   - Status: implemented.
   - Why: reviewers sometimes need to inspect `main..feature` or a commit range without checking out the right-hand ref and disturbing their current worktree.
   - Surface: `cr diff --range OLD..NEW` and `cr review --range OLD..NEW` reuse the normal summary/tree/hunk/JSON output while reading code outlines from `NEW`.
21. Large-review navigation sorting
   - Status: implemented.
   - Why: when many files change, humans often want to inspect risk files or largest churn first instead of reading Git order linearly.
   - Surface: `cr review --sort risk`, `--sort churn`, and `--sort path` reorder summary/detail/JSON output while the default `--sort git` preserves existing behavior.
22. Review session handoff
   - Status: implemented.
   - Why: reviewers often need to paste a compact change package into AI or team chat without manually trimming terminal tables and hunks.
   - Surface: `cr review --prompt` emits Markdown with review instructions, totals, per-file risk/purpose/focus metadata, and fenced compact diff hunks; it composes with sort, context, code, path, staged, base, and range controls.
23. Lightweight per-file extraction
   - Status: implemented.
   - Why: after scanning a large summary, reviewers need a fast way to expand exactly one file without copying a long path from the terminal.
   - Surface: `cr review` summary rows include `idx`, and `cr review --pick N` selects the Nth file after the current filters and sort; it composes with full terminal, `--summary`, `--json`, and `--prompt` output.
24. Lightweight review-state handoff
   - Status: implemented.
   - Why: large reviews often happen over several terminal passes; reviewers need a way to carry "already read" files forward without a database or IDE session.
   - Surface: `cr review --seen PATH` marks files as seen in terminal, JSON, and prompt output, and `--remaining` filters seen files after the current filters and sort.
25. Final daily-use audit
   - Status: implemented.
   - Why: before stopping, the tool should be checked against the original daily workflow instead of simply accumulating features.
   - Surface: verified local-change review, changed-file hierarchy, per-file hunks, file purpose, anchors, risk triage, large-review navigation, prompt handoff, review-state handoff, baseline/range review, and editable install with current tests and smoke commands.
26. In-session browse filtering
   - Status: implemented.
   - Why: once `cr browse` became the default TUI, large reviews still needed a fast way to jump from a long changed-file list to the relevant path without linear navigation.
   - Surface: `cr browse` supports `/` filter input, non-TTY `/query` and `filter query`, `c` / `clear`, active filter status, filtered numeric selection, filtered next/previous navigation, editor opening, and refresh selection clamping.
27. Browser module deepening
   - Status: implemented.
   - Why: continuing to add TUI behavior inside `src/cr/cli.py` made the CLI module shallow; the parser had to know about terminal raw keys, screen rendering, filter state, and editor handoff.
   - Surface: `src/cr/ui/browser.py` now owns the interactive browse module interface and implementation, while `src/cr/cli.py` delegates `cmd_browse` to `run_browser(args)`.
28. Package taxonomy
   - Status: implemented.
   - Why: once the project grew past a few files, placing Git, source parsing, review rendering, and terminal UI modules directly under `src/cr` made the package interface too abstract.
   - Surface: `src/cr` now exposes architecture through `vcs`, `source`, `review`, and `ui` packages, with `CONTEXT.md` documenting where new behavior should live.
29. Review workflow deepening
   - Status: implemented.
   - Why: `src/cr/cli.py` was still carrying review selection, seen/pick handling, JSON/prompt/terminal output branching, and file detail rendering after the package split.
   - Surface: `src/cr/review/workflow.py` owns `cr review` execution, while `src/cr/review/changes.py` owns reusable changed-file facts shared with `cr diff`.
30. Simple default command ergonomics
   - Status: implemented.
   - Why: daily use should be `cr` first, with optional browse filters only when needed; `cr --context 0` should not force users to remember the `browse` subcommand.
   - Surface: CLI prepends `browse` when the first argument is an option, so `cr --code`, `cr --sort risk`, and `cr --context 0` enter the interactive browser directly.
31. Shared review facts
   - Status: implemented.
   - Why: after `cr review` moved into `cr.review`, `cr browse` still carried duplicate change selection, sorting, hunk rendering, code-file detection, and modified-symbol rules.
   - Surface: `src/cr/ui/browser.py` now reuses `src/cr/review/changes.py` for review-scope facts and keeps only interaction state, terminal rendering, and editor opening behavior.
32. Stable browser screen layout
   - Status: implemented.
   - Why: once build output became an in-session background task, bottom-log updates and raw-key navigation started competing for the same terminal rows; this made the product feel unstable right when it should behave like a small IDE replacement.
   - Surface: `cr browse` now treats the screen as four layers: help/context, main content, background task panel, and input prompt. Build output stays in a 5-10 line task panel above the prompt, panel refresh does not clear the main screen, and raw-key navigation no longer emits extra newlines.
33. Build task controls
   - Status: implemented.
   - Why: a workbench cannot only start long-running builds; users need a way to stop a bad run, rerun after a fix, and read the outcome without leaving review context.
   - Surface: `cr browse` supports `: build`, `: stop` / `: cancel`, and `: rerun` / `: rebuild`. The build panel distinguishes running, stopping, stopped, succeeded, failed, failed-to-start, and idle states.
34. Review workspace scope clarity
   - Status: implemented.
   - Why: once the browser can inspect worktree, staged changes, commit history, and ref ranges, users need to know exactly which review workspace is active and switch between them without restarting.
   - Surface: `cr browse` displays `Scope: ...` for worktree, staged, all local changes, base refs, explicit ranges, recent commits, and selected commits. The command prompt supports `worktree`, `staged`, `all`, `base REF`, and `range OLD..NEW` and resets view-local selection/filter/scroll state on scope switches.
35. Browser command list
   - Status: implemented.
   - Why: once `cr browse` has many in-session commands, users need an in-tool way to discover them without reading README or triggering unknown-command noise.
   - Surface: `cr browse` supports `commands`, `cmds`, and `help commands`; in raw mode, `:` followed by empty input or `?` opens the same command list. The list groups navigation, review scope, build task, file, and session commands, and `b` returns to the file list.
36. Build process group lifecycle
   - Status: implemented.
   - Why: stopping only the build entry process can leave child build processes running, which risks continued log output after the user returns to normal review navigation.
   - Surface: interactive background builds run in an isolated process group. `: stop` / `: cancel` terminates that process group first, falls back to parent-process termination if group cleanup fails, and keeps the build panel's stopped/succeeded/failed lifecycle semantics stable.
37. Build stop hard-kill escalation
   - Status: implemented.
   - Why: graceful process-group termination is still only a request; real build tools may ignore or delay it, leaving the workbench stuck in stopping while logs continue.
   - Surface: `: stop` / `: cancel` records the stop request time. If the build is still running after a short grace period, `cr browse` sends one force-kill signal to the build process group, falls back to the parent process when no group id exists, and keeps duplicate escalation logs out of the bottom panel.
38. Browser workspace persistence
   - Status: implemented.
   - Why: a workbench should preserve a user's review context across terminal sessions instead of dropping them back at the default file list every time.
   - Surface: default `cr browse` saves scope, filter, selected path/index, and list/file layer under `.git/cr/browse-state.json` on clean exit. Later default sessions restore that workspace; explicit scope flags and pathspecs win over saved state.
39. Browser review progress markers
   - Status: implemented.
   - Why: when a review spans many files or a previous commit, users need a cheap way to know what has already been scanned without leaving the browser or maintaining a separate checklist.
   - Surface: `cr browse` supports `m` / `seen` / `done` to mark the selected file seen, `todo` / `unseen` / `unmark` to clear it, `remaining` to show only unreviewed files, and `allfiles` / `show all` to return to the full list. The file tree shows `[x]` / `[ ]`, file headers show `seen` / `todo`, and progress state persists in `.git/cr/browse-state.json`.
40. Browser frame rendering
   - Status: implemented.
   - Why: build logs, temporary prompts, navigation, and future task panels must not compete for terminal output ownership; otherwise the workbench feels unstable exactly when it is trying to replace IDE workflows.
   - Surface: raw-key `cr browse` now has a `BrowserFrame` render state. Full redraw records the active layout and task-panel snapshot; build panel partial refreshes only write while that frame is still valid; `:` and `/` temporary input force the next update to restore the full frame.
41. Executable command palette
   - Status: implemented.
   - Why: a workbench should not make users memorize every command once actions span review, build, scope switching, file operations, and session control.
   - Surface: raw-key `commands` mode is now a command palette. It shows executable actions grouped by purpose, supports ↑/↓ / j/k selection, Enter executes the selected command through the normal command path, and it avoids accidentally opening the selected review file from the command page.
42. Command palette search
   - Status: implemented.
   - Why: once the command palette becomes the action entry point, users need to type a few letters and narrow the action list instead of scanning every group.
   - Surface: raw-key `commands` mode supports `/` to filter executable commands by group, label, command, or description; `c` / `clear` clears the command filter without touching the file path filter; Enter executes the selected filtered command through the normal command path.
43. Task panel history
   - Status: implemented.
   - Why: build output should feel like an IDE task/output panel, not a transient log tail; after reruns users still need to see whether the previous build succeeded, failed, or stopped.
   - Surface: the browser keeps session-local `TaskRecord` history for completed tasks. The bottom task panel shows compact recent task results alongside current task status and latest logs, records each completed task once, retains history across reruns, and keeps the history out of `.git/cr/browse-state.json`.
44. Browser layer ownership
   - Status: implemented.
   - Why: the product is moving toward a terminal IDE replacement, so page regions need ownership rules. Build logs, command feedback, and navigation must not all write to stdout independently.
   - Surface: raw-key `cr browse` treats the page as context/status, main content, background task panel, and input prompt layers. Open-file feedback, invalid choices, and unknown commands render in the context/status layer; they mark the frame dirty so build output waits for a full redraw instead of patching a stale bottom panel.
45. Product navigation breadcrumbs
   - Status: implemented.
   - Why: the product hierarchy is `Review Scope -> Changed Files -> File Detail`, but users need to see that hierarchy in the interface, not only in docs.
   - Surface: the context/status layer now renders breadcrumbs such as `Scope: worktree > Files`, `Scope: commit abcdef12 > Files`, and `Scope: worktree > Files > src/Sample.ts`. Recent commits stay as `Scope: recent commits` because that screen is still a scope picker, not a file list.
46. Scope home
   - Status: implemented.
   - Why: Review Scope is the product's first layer; users should not have to memorize scattered commands to find worktree, staged, all local changes, commits, base, and range entry points.
   - Surface: `scopes` / `scope` opens a Review Scope Home. Users can select worktree, staged, all local changes, or recent commits with keyboard navigation and Enter. Base/range are shown as parameterized command hints, preserving existing `: base REF` and `: range OLD..NEW` flows.
47. Task command breadth
   - Status: implemented.
   - Why: replacing IDE-like daily workflow needs more than build; users need to run tests and lint without leaving review context.
   - Surface: `cr browse` supports `: build`, `: test` / `: tests`, and `: lint` in the same bottom task panel. `: stop` / `: cancel` stop the current task, and `: rerun` / `: rebuild` rerun the most recent task kind. Build keeps the DouyinHarmony default; test/lint use explicit `--test-cmd` / `CR_TEST_CMD` and `--lint-cmd` / `CR_LINT_CMD`.
48. Task state naming
   - Status: implemented.
   - Why: after Task Panel started handling build/test/lint, build-specific runtime names made the architecture harder to extend and easier to misunderstand.
   - Surface: the current background task runtime is now `TaskState`, `BrowserState.task`, `_poll_task`, `_record_completed_task`, and `_task_panel_*`. Build-specific naming remains only where behavior is truly build-specific, such as `_build_command`.
49. Browser page model
   - Status: implemented.
   - Why: the product hierarchy is Review Scope, Changed Files, File Detail, and cross-layer overlays, but the browser implementation still spoke mostly in raw `mode` strings such as `list`, `file`, and `commands`.
   - Surface: `BrowserPage` now names Scope Home, Commit Picker, Changed Files, File Detail, and Command Palette. `BrowserState.page` is the primary state field, while `mode` remains a compatibility property and persisted workspace values stay unchanged.
50. Navigation module deepening
   - Status: implemented.
   - Why: after page naming became explicit, the browser main loop still owned many raw page transition assignments and reset rules, making back/open/next behavior harder to change safely.
   - Surface: `BrowserNavigation` now owns page transition intent and local reset rules for Changed Files, File Detail, Scope Home, Commit Picker, and Command Palette. It does not load Git data, switch review scopes, render output, or implement a real page stack.
51. ReviewWorkspace deepening
   - Status: implemented.
   - Why: replacing IDE-like workflows needs a stable review workspace model; active scope, changed files, selected commit, previous scope, filter/progress, and persisted selected path were still coordinated directly by `browser.py`.
   - Surface: `ReviewWorkspace` now owns active Review Scope state, changed-file loading, filter/progress state, selected file state, selected commit, previous scope, and workspace-state data mapping. Browser file I/O and rendering remain outside the workspace module.
52. Command dispatch deepening
   - Status: implemented.
   - Why: after navigation and workspace behavior moved behind named modules, the browser main loop still recognized command strings directly, making command aliases and parameter parsing easy to duplicate or accidentally change.
   - Surface: `BrowserCommandAction` and `parse_browser_command` now translate raw-key aliases, line-mode commands, parameterized commands, numeric selections, and unknown input into stable browser actions before `run_browser` executes them.
53. Command action execution deepening
   - Status: implemented.
   - Why: once command parsing had a seam, the browser main loop still owned every parsed action branch, so each new workbench operation would continue to increase one high-risk loop.
   - Surface: `BrowserCommandExecutor` now executes parsed browser actions and returns `BrowserActionResult` loop control. `run_browser` keeps input prompts, sentinels, workspace save-on-exit, and redraw scheduling, but no longer owns the full action branch table.
54. Task runtime extraction
   - Status: implemented.
   - Why: build/test/lint already behave like IDE task panel operations, but command resolution, process lifecycle, output capture, stop escalation, rerun, and history were still owned by `browser.py`.
   - Surface: `cr.ui.tasks` now owns Task Panel runtime behavior and `TaskState` / `TaskRecord`; `browser.py` uses that runtime while keeping terminal layout and panel rendering.

## Next P0 candidates

The long-term product direction is a terminal workbench for high-frequency IDE-like operations. Current next P0 candidates should come from real usage friction in this order:

1. Task configuration presets: add project-local task preset discovery for build/test/lint now that runtime behavior has a module home.
2. Real page stack: later consider explicit back/forward history if the current hierarchy-aware `back` behavior stops being enough.
3. File action breadth: consider copy path, open changed line, and reveal in editor actions once command/action/runtime modules are stable.
