## 为什么

`cr browse` 的目标已经从“能看 diff”扩展成终端里的轻量 IDE 工作台。当前 build 输出已经能固定在底部面板，但 raw-key 模式下仍有若干操作直接 `print()` 到 stdout，例如打开文件成功、无效数字、未知命令等。这些输出绕过了固定 frame，用户在 build 后继续操作时会看到页面被打乱。

同时，产品文档虽然提到四层布局，但代码和验收还没有把“每一层由谁拥有、哪些路径允许写终端”讲清楚。继续添加 build/test/lint/command palette 之前，需要先稳定页面层级所有权。

## 改什么

- 明确 `cr browse` raw-key 页面分为四层：上下文/状态层、主内容层、后台任务面板层、输入提示层。
- raw-key 模式下，普通操作反馈不得直接打印到 stdout，而是进入上下文/状态层并触发完整 redraw。
- build 面板局部刷新只能更新后台任务面板层；如果 frame 不完整、布局变化或状态层有新消息，必须拒绝局部刷新并等待完整 redraw。
- 将“页面层级模型”写入设计文档，作为后续 IDE-like 操作的基础。

## 不做

- 不引入第三方 TUI 框架。
- 不重写整个浏览器渲染循环。
- 不在本轮增加 test/lint 新任务类型。
- 不实现鼠标、split pane 或语言服务。
